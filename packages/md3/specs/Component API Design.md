# 컴포넌트 API 설계 철학
본 문서는 MD3 라이브러리 컴포넌트의 인터페이스 설계 원칙을 정의한다. 모든 컴포넌트는 사용자의 개발 경험을 극대화함과 동시에 Material Design 스펙의 무결성을 보장하도록 설계되어야 한다. 이를 위해 투 트랙 전략을 기본 아키텍처로 채택한다.

# 투 트랙 설계 전략
라이브러리는 통제된 완제품과 해체된 부품을 동시에 제공한다. 메인 컴포넌트는 제한된 속성만을 사용하여 디자인 시스템의 일관성을 강제한다. 특수한 렌더링이 필요한 극한의 상황을 대비하여 컴포넌트를 구성하는 뼈대 부품들을 네임스페이스 하위에 개별적으로 노출한다. 사용자는 이를 합성 컴포넌트 패턴으로 직접 조립하여 프레임워크의 구조적 한계를 우회할 수 있다.

```tsx
<IconButton icon="search" />

<IconButton.Root>
  <IconButton.Container />
  <IconButton.Icon>search</IconButton.Icon>
  <IconButton.StateLayer />
</IconButton.Root>
```

# 속성과 자식 노드의 역할 분리
컴포넌트의 가장 핵심이 되는 데이터는 HTML 마크업 관례에 따라 자식 노드 공간에 배치한다. 버튼의 라벨과 같은 주 콘텐츠가 이에 해당한다. 아이콘과 같은 보조적인 시각 요소는 반드시 명시적인 속성으로 분리하여 전달받아야 한다. 이는 메인 자식 노드의 공간을 안전하게 확보하여 태그 교체 패턴을 원활하게 지원하기 위함이다.

```tsx
<Button icon="search">검색</Button>
```

# 구조적 변형의 1차 방어
현업에서 빈번하게 발생하는 구조적 변형은 합성 컴포넌트 사용을 강제하지 않고 속성 선에서 방어해야 한다. 우측 아이콘 배치와 같이 디자인 스펙에 정의된 표준적인 레이아웃 변형은 별도의 위치 지정 속성을 통해 메인 컴포넌트 내부에서 즉시 처리되어야 한다. 부품을 직접 조립하는 행위는 속성만으로 해결할 수 없는 예외적인 상황으로 제한한다.

```tsx
<Button trailingIcon="arrow_right">다음</Button>
```

# 다형성과 타입 안정성
컴포넌트의 렌더링 태그 변경은 타입 복잡성을 낮추기 위해 자식 요소와의 병합 방식을 사용한다. 이때 자식 노드의 허용 여부는 태그 변경 속성의 상태에 따라 엄격하게 분기되어야 한다. 태그를 교체하는 경우 자식 노드에는 병합될 단일 리액트 요소가 강제되며 기본 상태에서는 의도치 않은 텍스트 주입을 타입 레벨에서 원천 차단한다.

```tsx
<IconButton icon="search" asChild>
  <Link href="/search" />
</IconButton>
```
